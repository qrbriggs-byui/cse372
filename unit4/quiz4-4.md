# Quiz: Understanding Common and Complex Formats for Acceptance Criteria

This quiz tests your understanding of the different formats and uses of acceptance criteria in software development.

---

**1.** What is the primary purpose of acceptance criteria in software development?  
A. To document visual design guidelines  
B. To define what must be true for a story to be considered complete  
C. To measure system performance speed  
D. To replace user stories entirely  
**→ Correct answer:** B  

---

**2.** Which of the following best describes the **Checklist format** for acceptance criteria?  
A. A narrative story about user behavior  
B. A structured testing language using “Given–When–Then”  
C. A simple list of conditions that must be met for the story to be complete  
D. A document that outlines all user interviews  
**→ Correct answer:** C  

---

**3.** In the study room booking example, which of the following is a valid checklist item?  
A. “The system should always look nice.”  
B. “The user can select a date and time.”  
C. “Users should like the interface.”  
D. “The feature should work with all devices.”  
**→ Correct answer:** B  

---

**4.** The **Given–When–Then format** is most closely associated with which development approach?  
A. Test-Driven Development (TDD)  
B. Behavior-Driven Development (BDD)  
C. Extreme Programming (XP)  
D. Rapid Application Development (RAD)  
**→ Correct answer:** B  

---

**5.** In Gherkin syntax, what does the **Given** statement describe?  
A. The expected outcome of a test  
B. The initial context or preconditions for a scenario  
C. The specific user action being tested  
D. The visual appearance of the interface  
**→ Correct answer:** B  

---

**6.** What do the **When** and **Then** parts of Gherkin syntax represent?  
A. User goals and motivations  
B. User actions and expected outcomes  
C. System limitations and constraints  
D. Visual designs and color schemes  
**→ Correct answer:** B  

---

**7.** Why might a team use **complex acceptance criteria** with multiple conditions joined by “And”?  
A. To describe multiple test cases in one sentence  
B. To represent features that have several dependent rules or outcomes  
C. To make the documentation more creative  
D. To reduce collaboration between testers and developers  
**→ Correct answer:** B  

---

**8.** When should **simple acceptance criteria** (like checklists) be used?  
A. When a feature involves multiple user states and integrations  
B. When a story is straightforward and can be tested manually  
C. When the product owner prefers longer documentation  
D. When the design phase has not started yet  
**→ Correct answer:** B  

---

**9.** Which of the following is an example of a **complex acceptance criterion** using Gherkin syntax?  
A. “The system displays a confirmation message.”  
B. “Given the user is logged in and has no conflicts, when they confirm booking, then the system sends both an in-app and email confirmation.”  
C. “The user can access the home screen.”  
D. “The page loads in under two seconds.”  
**→ Correct answer:** B  

---

**10.** According to the reading, what is the key to choosing between simple and complex acceptance criteria?  
A. Balancing clarity and efficiency based on the complexity of the feature  
B. Using complex criteria for all user stories to avoid misunderstandings  
C. Always preferring checklists because they’re faster to write  
D. Writing only Gherkin syntax for consistency  
**→ Correct answer:** A
